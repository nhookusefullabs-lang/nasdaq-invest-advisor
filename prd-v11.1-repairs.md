# prd-v11.1-repairs.md — 측정 인프라 수리 (Ralph 호환 태스크 문서)

> 배경: v11 실데이터 검증(2026-07)에서 측정 인프라 결함·공백 4건이 확인됐다.
> ① 눌림목 관찰 조건 표본 고사 (체결률 0.2~1.3%, 전 구간 체결 1~3건 — immediate조차 0.2%)
> ② exit_structural 발동률 0% (480건×60일에서 구조 손절 미발동 — 비현실적)
> ③ exit_climax_partial 미기록 (청산 E 미측정 — 부분 포지션 인프라 미배선)
> ④ 청산 변형의 국면별 분해 부재 (청산 A의 선커밋 판정 불가의 직접 원인)
> **v11.1은 수리 릴리스다. 새 가설·새 변형·조건 완화는 넣지 않는다.**
> 루프 규칙: 매 이터레이션마다 미완료 스토리 중 가장 위의 것 하나만 구현하고,
> 승인 기준을 전부 통과하면 체크박스를 채우고 커밋한 뒤 이터레이션을 종료한다.

## 프로젝트 컨텍스트 (매 이터레이션 필독)

- v11 자산 위에서 작업. 절대 원칙 전부 유지 (같은 코드·미래 참조 불가·자동 최적화 금지·
  adopted=false·Out만 표시·graceful degradation·신규 npm 의존성 금지)
- **수리의 경계**: 루프가 고치는 것은 "스펙과 구현의 불일치(버그)"만이다.
  스펙 자체(상수·조건 정의)가 원인으로 판명되면 **고치지 말고** 진단 결과를 progress.txt에
  기록하고 다음 스토리로 넘어간다 — 조건 완화는 운영자 결정(해석 세션)이다.
  스펙 기준 문서: design-entry-point-engine.md, PRD_Nasdaq11 §4.2~4.3
- US-5의 배지 문구 조정은 운영자가 승인한 표시 변경이다 (v11 해석 세션 근거)
- 품질 게이트 (모든 스토리 공통): `npm run test` 전체 통과 + `npm run build` 성공

## 유저 스토리 (우선순위 순 — 위에서부터 하나씩)

### US-1. 눌림목 관찰 조건 — 퍼널 진단 + 버그 수리
- [x] **운영자로서, P1~P4 중 어느 조건이 신호를 고사시키는지 단계별 통과 수로 보고 싶다.**

**구현:** `scripts/backtest.mjs` + `src/lib/pullback.js` 점검
- **퍼널 계측**: 평가일마다 P1 통과 수 → ∩P2 → ∩P3 → ∩P4 단계별 카운트를 집계해
  backtest.json `pullbackFunnel`(sample×국면별)과 콘솔에 출력
- **재현 픽스처로 스펙 대조**: "명백히 건강한 눌림" 합성 시계열(템플릿 통과 + 피벗 −15% +
  SMA200 위 + 눌림 거래량 60% 수준)을 만들어 관찰 조건이 발화해야 함을 테스트로 고정 —
  실패하면 구현 버그이므로 스펙에 맞게 수리 (특히 P4의 "직전 상승 구간(피벗 이전 동일 길이)"
  창 구성과 P2의 피벗 참조가 설계 문서와 일치하는지, `pullback_immediate`의 fillRate가
  관찰 충족 시 1.0이 되는지)
- 수리 후에도 실데이터 퍼널에서 고사가 스펙 수준(조건이 원래 좁음)으로 판명되면:
  결박 조건과 단계별 수치를 progress.txt에 기록하고 **조건은 변경하지 않는다**

**승인 기준:**
1. 합성 픽스처에서 관찰 조건 발화 + immediate fillRate = 100% (스펙 동작 보증)
2. 퍼널 카운트의 단조 감소·합산 정합성 테스트 (P1 ⊇ ∩P2 ⊇ ∩P3 ⊇ ∩P4)
3. pullbackFunnel이 스키마 검증을 통과하고 콘솔 표가 출력된다
4. 버그가 있었다면 수리 내용을, 스펙 문제라면 결박 진단을 progress.txt에 기록

### US-2. exit_structural 발동 0% 수리
- [x] **운영자로서, 구조 손절이 이탈 시 실제로 발동하는 올바른 측정을 원한다.**

**구현:** `scripts/lib/exits.mjs` 점검
- 재현 픽스처: 진입 후 종가가 피벗×0.97을 명백히 하회하는 경로 → 당일 종가 청산이
  발생해야 함을 테스트로 고정. 눌림목형(눌림 저점×0.98)도 동일
- 유력 점검 지점: 손절선 산출의 피벗 참조(진입 시점 고정 여부), 진입 유형 매핑의 기본값
  (유형 불명 시 손절선 미정의로 전부 스킵되는 경로), 비교 부호
- 수리 후 실데이터 발동률을 콘솔에 출력 (판정 기준 <40%와의 대조는 운영자 몫)

**승인 기준:**
1. 이탈 경로 픽스처에서 발동 (돌파형·눌림목형 각각) + 미이탈 경로에서 60일 만기 (경계 테스트)
2. 손절선이 진입 시점에 고정됨을 보증하는 테스트 유지
3. 기존 청산 변형 결과 불변 (수리의 부수 효과 없음 — 회귀)

### US-3. exit_climax_partial 배선 (청산 E 측정 복구)
- [x] **운영자로서, 클라이맥스 부분 청산이 variants에 정상 기록되길 원한다.**

**구현:** `scripts/lib/exits.mjs` + variants 등록
- `exit_climax_partial`: X4 발생 시 50% 청산·잔여 50% 60일 만기 — US-3(v11)의 부분 포지션
  인프라(performance.mjs)와 실제 배선. exitSignals.js의 X4 재사용 (재구현 금지)
- variants[]에 outDetail(발동률·평균 보유일 포함)로 기록, 3자 비교(무청산/전량/부분)를 note에

**승인 기준:**
1. X4 발생/미발생 경로 픽스처에서 부분 청산 수익률이 가중 평균 손계산과 일치
2. 산술 새니티: 부분 청산 성과가 무청산과 전량 청산 사이 (기존 테스트 문법 재사용)
3. variants에 레코드 존재 + adopted=false

### US-4. 청산 변형의 국면별 분해 (regimeDetail)
- [x] **운영자로서, 청산 A의 선커밋 기준("하락 국면 MDD −5%p")을 판정할 재료를 원한다.**

**구현:** `scripts/backtest.mjs` 집계 확장
- 모든 청산 변형·조합 레코드에 `regimeDetail[]` 추가: 신호일 국면(상승/중립/하락) ×
  {signals, winRate, avgExcess, medianExcess, mdd, stopHitRate, avgHoldingDays}
- 기준선(고정 보유)의 국면별 값도 병기해 델타 계산이 가능하게
- 스키마 v4 내 optional 필드로 (로더 하위 호환 유지 — UI 무변화)

**승인 기준:**
1. regimeDetail 합산 정합성 (국면별 신호 수 합 = 전체)
2. 국면 귀속이 신호일 슬라이스 기준 (미래 참조 불가 테스트)
3. 무효 픽스처(regime enum 오류) 거부 + 로더 하위 호환 테스트
