# PRD: 나스닥 종목추천 v6 — 리서치 서브에이전트 (계획 문서)

> 이 문서는 v5(as-built)까지 배포된 메인시스템에 **리서치 서브에이전트**를 추가하기 위한 설계 문서다.
> v5까지의 메인시스템은 변경 최소화가 원칙이며, 서브에이전트는 기존 `nasdaq100.json`과 동일한
> **정적 스냅샷 패턴**을 따르는 두 번째 데이터 파일(`research.json`)을 생성하는 독립 구성요소다.

## 1. 개요

| 항목 | 내용 |
|---|---|
| 구성요소명 | 나스닥100 리서치 서브에이전트 |
| 한 줄 정의 | 추천 상위 종목에 대해 최신 뉴스·실적·촉매 정보를 자율 리서치하여 `research.json` 스냅샷으로 제공하는 Claude Code 기반 에이전트 |
| 목적 | 기술적 지표(정량) 기반 추천에 **정성적 맥락**(왜 오르는가/어떤 리스크가 있는가)을 보강하여 초보 투자자의 이해를 돕는다 |
| 메인시스템과의 관계 | 단방향 데이터 공급자. 메인 SPA는 `research.json`을 fetch하여 표시만 하고, 서브에이전트의 존재를 몰라도 동작에 지장 없음 (graceful degradation) |

## 2. 배경 및 문제 정의

- v5 추천 화면은 "RSI 62, MACD 골든크로스, 주도 섹터 소속" 같은 **지표 사유**만 제공한다.
- 초보 투자자는 지표 신호만으로는 "이 종목에 지금 무슨 일이 일어나고 있는지"를 알 수 없다.
- 실시간 뉴스 API 연동(백엔드 필요)은 Won't have 범위이므로, **주기적 스냅샷 리서치**로 절충한다.

## 3. 범위

### Must have
1. **리서치 대상 대화형 선정** — 서브에이전트 실행 시점에 두 단계로 대상을 확정한다:
   - (a) **추천 풀 제시 후 선택**: 현재 `nasdaq100.json` 기준 추천 로직 실행 결과(상위 10개 + 고득점 특별 편입)를 점수·사유와 함께 보여주고, 운영자가 그중 리서치할 종목을 선택 (전체 선택 가능)
   - (b) **관심 종목 직접 추가**: 추천 여부와 무관하게 운영자가 원하는 티커를 직접 입력해 추가. 단 `nasdaq100.json`에 존재하고 데이터가 충분한(dataSufficient) 종목으로 한정 — 유니버스 밖 티커는 사유와 함께 거부
   - 1회 세션 대상은 **최대 15종목** 권장 (토큰 예산, §7-1)
2. **종목별 자율 리서치 (AutoResearch 패턴)** — 웹검색 기반으로 종목당 다음 항목 수집 (§4.5 우선 소스 목록 활용):
   - 최근 주가 흐름의 배경 요약 (2~3문장)
   - 긍정 촉매 (catalysts) 최대 3개
   - 리스크 요인 최대 3개
   - 기관투자자 동향 (13F 기반, 확인 가능한 경우)
   - 종합 센티먼트 (positive / neutral / negative)
   - 출처 URL 목록
3. **`research.json` 스냅샷 생성** — 아래 §4.2 스키마 준수, `public/data/`에 저장
4. **메인 SPA 통합 (화면2)** — 추천 카드에 "AI 리서치" 접이식 섹션 표시. `research.json`이 없거나 해당 티커가 없으면 섹션 자체를 숨김
5. **기술적 추천과 시각적 분리** — 리서치 섹션에 `researchedAt` 표시 + "AI가 수집한 참고 정보이며 투자 판단의 근거가 아닙니다" 문구 고정

### Should have
- 리서치 실패 종목 스킵 및 사유 로그 (기존 exclusion_log 패턴 재사용)
- 센티먼트 배지 색상 표시 (positive=빨강 계열, negative=파랑 계열 — 국내 관례, 미니차트와 일관)

### Won't have (이번 범위 제외)
- 실시간 리서치 / 백엔드 연동 — **배포된 웹 화면에서 리서치를 요청하는 기능은 불가** (백엔드 없음). 대상 선택은 서브에이전트 실행 시점의 대화형 선택으로만 가능
- 100종목 전체 리서치
- 포트폴리오 화면 통합 (화면2·3만)
- 나스닥100 유니버스 밖 종목 리서치
- 리서치 이력 누적 보관 (스냅샷 1개만 유지, 덮어쓰기)
- 자동 스케줄 실행 (수동 실행으로 시작, 자동화는 확장 아이디어)

## 4. 핵심 기능 상세

### 4.1 리서치 파이프라인 (실행 흐름)

```
[1] nasdaq100.json 로드
[2] 기존 추천 로직(lib/recommend.js와 동일 기준) 실행
    → 추천 풀(상위 10 + 고득점 편입)을 점수와 함께 운영자에게 제시
    → 운영자가 풀에서 선택 + 관심 종목 티커 직접 추가 (유니버스 내 검증)
    → 최종 리서치 대상 목록 확정
[3] 티커별 AutoResearch 루프:
    검색 → 수집 → 요약 → 스키마 검증 → 누적
[4] research.json 생성 (전체 원자적 쓰기 — 부분 실패 시 기존 파일 유지)
[5] 수동 검토 후 커밋 → GitHub Pages 재배포
```

- 실행 주체: Claude Code (로컬, 수동 실행)
- 실행 주기: 주 1회, **데이터 수집(collect_data.py)과 같은 날(토요일) 데이터 수집 직후** 실행
  — 리서치 대상 선정이 최신 지표 기준이 되도록 순서 고정

### 4.2 research.json 스키마 (버전 1)

```json
{
  "schemaVersion": 1,
  "researchedAt": "YYYY-MM-DD",
  "basedOnDataOf": "YYYY-MM-DD",
  "items": [
    {
      "ticker": "AAPL",
      "sentiment": "positive",
      "summary": "최근 상승 배경 2~3문장 요약",
      "catalysts": ["촉매1", "촉매2"],
      "risks": ["리스크1", "리스크2"],
      "institutionalActivity": "1분기 13F 기준 주요 기관 동향 요약 (확인 불가 시 null)",
      "analystView": "수집 시점 기준 애널리스트 컨센서스 요약 (TipRanks/Zacks, 확인 불가 시 null)",
      "sources": [
        { "title": "기사 제목", "url": "https://...", "date": "YYYY-MM-DD", "operatorProvided": false }
      ],
      "signalPassed": true,
      "origin": "recommended"
    }
  ],
  "skipped": [
    { "ticker": "XXXX", "reason": "검색 결과 부족" }
  ]
}
```

- `basedOnDataOf`: 리서치 대상 선정에 사용한 `nasdaq100.json`의 `generatedAt` 값 — 두 스냅샷의 정합성 추적용
- `origin`: `"recommended"`(추천 풀에서 선택) 또는 `"userRequested"`(관심 종목 직접 추가) — 선정 경위를 기록하고 화면 표시를 구분하는 근거. `userRequested` 항목은 `signalPassed`가 없을 수 있음
- `sources`: 최소 1개 필수. 출처 없는 항목은 `skipped` 처리
- `summary`·`catalysts`·`risks`는 **출처에 근거한 사실 요약**만 담고, 매수/매도 권유 표현("사야 한다", "유망하다" 등) 금지 — 프롬프트 규칙으로 강제

### 4.3 메인 SPA 통합 (화면2 + 화면3 변경 사항)

**공통 컴포넌트** (`ResearchSection.jsx`) — 접이식 "AI 리서치" 섹션:
- 접힌 상태(기본): 센티먼트 배지 + summary 첫 문장
- 펼친 상태: 전체 summary, 촉매/리스크 목록, 출처 링크, `researchedAt`
- `origin: "userRequested"` 항목에는 "관심 종목 리서치" 배지 표시 (추천 기반 리서치와 구분)

**표시 위치:**
- **화면2 (추천 결과)**: 각 추천 카드 하단 — 해당 티커가 `research.json`에 있을 때만
- **화면3 (시뮬레이션)**: 각 종목 카드 하단 — v5에서 시뮬레이션 화면에 관심 종목을 직접 추가할 수 있게 되었으므로, 추천 여부와 무관하게 리서치된 관심 종목을 확인하는 자연스러운 위치

**안전장치:**
- `research.json` fetch 실패 또는 티커 미존재 시: 섹션 미표시 (에러 노출 없음)
- `basedOnDataOf` ≠ 현재 `generatedAt`이면 "이 리서치는 이전 데이터 기준입니다" 경고 표시

### 4.4 우선 리서치 소스 (Priority Sources)

리서치 시 일반 웹검색에 앞서 아래 무료 소스를 우선 조회한다. 소스는 **접근 방식에 따라 2계층**으로 나뉜다.

**[계층 A] 자동 조회 소스 — 에이전트가 공개 페이지를 직접 검색·조회**

| 소스 | 제공 정보 | 활용 방식 | 주의사항 |
|---|---|---|---|
| **SEC EDGAR (13F 보고서)** | 기관투자자 분기별 보유 현황 | 대상 종목에 대한 주요 기관의 신규 매수/청산/증감 확인 → `institutionalActivity` 필드에 요약 | **분기 종료 후 45일 시차** — "최신 동향"이 아니라 "직전 분기 동향"임을 화면에 명시 |
| **Dataroma** | 유명 가치투자자(슈퍼인베스터)들의 13F 집계 | EDGAR 원문보다 빠르게 "누가 이 종목을 담았는가" 파악 | 13F 기반이므로 동일한 시차 존재. 집계 사이트이므로 원출처(EDGAR) 교차 확인 권장 |
| **Macrotrends** | 장기 재무 추이 (매출, 마진, PER, 부채 등) | 촉매/리스크의 근거 보강 (예: "마진 개선 추세", "부채 증가") | 기술적 지표(단기)와 시계가 다른 펀더멘털(장기) 정보 — summary에서 혼동 없게 구분 서술 |
| **TipRanks** | 애널리스트 컨센서스, 목표주가, 스마트스코어 | 공개 범위의 컨센서스 등급·목표주가를 `analystView`에 요약 | 상세 데이터는 회원 전용 — 공개 범위만 사용, 접근 실패 시 스킵 |
| **Zacks Investment Research** | Zacks Rank(1~5), 실적 전망 수정 방향 | 공개 범위의 Rank·실적 전망을 `analystView`에 반영 | Rank 산출 상세는 유료 — 등급 자체와 방향성만 인용 |
| **The Motley Fool** | 종목 분석 기사 (대체로 공개) | 촉매/리스크 서술의 근거 기사로 활용 | 성장주 낙관 편향 성향 — 단일 근거로 쓰지 않고 교차 확인 |
| **초이스스탁US** | 국내 무료 미국주식 정보 (재무 요약, 적정주가 등) | 한국어 요약·펀더멘털 교차 확인 | 적정주가 등 자체 산출값은 방법론 불명 — 참고 수준으로만, 수치 단독 인용 지양 |
| 일반 웹검색 (뉴스) | 최근 뉴스·실적 발표·이벤트 | 위 소스에서 커버 못 하는 최신 촉매·리스크 보완 | 출처 신뢰도 확인 (원출처 우선) |

**[계층 B] 회원 콘텐츠 소스 — 운영자 수동 제공 (human-in-the-loop)**

| 소스 | 제공 정보 | 활용 방식 |
|---|---|---|
| **Value Investors Club (VIC)** | 심층 투자 아이디어 write-up | 게스트 열람(약 90일 지연) 범위는 계층 A처럼 시도, 회원 전용 글은 계층 B 절차 |
| **Seeking Alpha** | 뉴스·분석 기사 | 강한 페이월+봇 차단으로 자동 접근 사실상 불가 — 계층 B 절차 전용 |
| **IBD50 (Investor's Business Daily)** | 성장주 50선 리스트 | 리스트 자체가 구독 콘텐츠 — 대상 종목의 IBD50 편입 여부를 운영자가 확인해 입력 |

**계층 B 절차:** 리서치 세션 중 에이전트가 해당 종목의 계층 B 소스 확인을 운영자에게 요청 →
운영자가 **본인 계정으로 직접 로그인·열람**한 뒤 핵심 내용(등급, 논거, 편입 여부 등)을 세션에 요약 입력 →
에이전트는 이를 `sources[]`에 `operatorProvided: true`와 열람일·URL과 함께 기록.
**에이전트는 로그인 계정 정보(아이디/비밀번호)를 절대 받지 않으며, 로그인 벽 우회 자동화를 시도하지 않는다** (이용약관 준수).

**운영 원칙:**
- 소스별 접근 실패(로그인 벽, 차단, 구조 변경)는 오류가 아니라 **스킵 후 대체 소스 진행** — 소스 하나 실패로 종목 전체를 skipped 처리하지 않는다
- 13F·Dataroma 기반 정보는 시차가 크므로, 최근 뉴스와 시점이 어긋나는 경우 summary에서 시점을 명시한다 (예: "1분기 기준 ○○ 펀드 신규 편입")
- 애널리스트 등급류(TipRanks/Zacks)는 **수집 시점 기준**임을 `analystView`에 함께 표기한다
- 계층 B 입력이 없어도 종목 리서치는 계층 A만으로 완결 가능해야 한다 (계층 B는 보강용)
- 모든 소스는 `sources[]`에 URL·날짜와 함께 기록한다. 유료·회원 콘텐츠 원문은 복사하지 않고 재서술 요약만 한다

### 4.5 개발 도구 활용 계획 (강의 고급기능 매핑)

| 단계 | 도구 | 용도 |
|---|---|---|
| 서브에이전트 구현 | **Ralph 루프** 또는 **Goal** | 본 PRD를 태스크 목록으로 변환 후 자율 구현. 태스크 수가 적으므로(4~5개) Goal 단발 실행도 가능 |

| 통합 검증 | 기존 vitest | research.json 스키마 검증 테스트 추가 |

## 6. 기술 스택

- 서브에이전트: Claude Code + 웹검색 도구 (별도 서버·API 키 불필요)
- 산출물: 정적 JSON (기존 패턴 동일)
- 메인 SPA: React 컴포넌트 1~2개 추가 (`ResearchSection.jsx` 등), 차트 라이브러리 등 신규 의존성 없음

## 7. 비기능 요구사항

1. **토큰 예산**: 종목당 검색 3~5회 상한, 대상 최대 15종목 → 1회 리서치 세션이 Max 플랜 5시간 창을 과도하게 소모하지 않도록 설계. 실행 전 `/usage` 확인 습관화
2. **비결정성 관리**: 동일 조건 재실행 시 문구가 달라질 수 있음을 전제 — 스냅샷 파일과 `researchedAt`으로 "이 시점의 리서치"임을 명시
3. **원자성**: 리서치 도중 실패해도 기존 `research.json`은 훼손되지 않음 (전체 완료 후에만 교체)
4. **면책**: 기존 §9 면책 문구와 별개로 리서치 섹션 전용 문구 추가 (§3 Must-5)
5. **저작권**: summary는 출처 기사 원문을 복사하지 않고 재서술. 인용은 출처 링크로 대체

## 8. 성공 기준

- [ ] 서브에이전트 1회 실행으로 유효한 `research.json`이 생성된다 (스키마 검증 통과)
- [ ] 실행 시 추천 풀이 점수와 함께 제시되고, 운영자가 그중에서 선택할 수 있다
- [ ] 추천 풀에 없는 관심 종목을 티커 입력으로 추가할 수 있고, 유니버스 밖 티커는 거부된다
- [ ] 모든 리서치 항목에 출처가 1개 이상 포함되고 `origin`이 올바르게 기록된다
- [ ] 우선 소스(EDGAR/Dataroma/Macrotrends/VIC) 중 접근 가능한 소스가 실제 조회되고, 13F 기반 정보에는 기준 분기가 명시된다
- [ ] 화면2·3에서 리서치 섹션이 표시되고, research.json이 없어도 앱이 정상 동작한다
- [ ] 관심 종목 리서치는 "관심 종목 리서치" 배지로 추천 기반 리서치와 구분된다
- [ ] 매수/매도 권유 표현이 산출물에 포함되지 않는다
- [ ] `basedOnDataOf` 불일치 시 경고가 표시된다
