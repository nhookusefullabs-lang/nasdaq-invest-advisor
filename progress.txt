## Iteration 4 — US-4. 신규 지표 함수 2차 (스토캐스틱 + ATR + OBV)
- 완료 스토리: US-4
- 변경 파일: src/lib/indicators.js (stochastic, atr, atrPercent, obv + private helper
  smaAllowingNulls 추가), src/lib/indicators.test.js (10개 테스트 추가)
- 설계 결정: obv()는 다른 v7 신규 지표(스냅샷 반환)와 달리 **배열**을 반환함 — PRD 명세
  그대로("누적 OBV 배열"), 필터가 "OBV vs OBV의 SMA20"을 비교하려면 과거 흐름 전체가
  필요하기 때문. stochastic은 분모 0(고가=저가)이 발생한 날의 Fast %K를 null로 두고,
  smaAllowingNulls 헬퍼로 그 null이 평활(SMA) 구간에 전파되게 함 — 억지로 스킵해
  결측을 감추지 않음.
- 테스트 수 변화: 111 → 121 (+10), 전체 통과. npm run build 통과.
- 운영자 액션 필요: 없음
- 다음 스토리 메모: US-5 (constants.js 임계 상수 + filters.js 판정 함수 5종.
  ATR 백분위는 유니버스 dataSufficient 종목 전체 분포 기준 — 개별 종목 함수가 아니라
  풀 단위 계산이 필요함에 유의. 52주 필터는 제외 종목 수도 함께 반환해야 함).

- 설계 결정: 판정 함수는 티커 전체 객체가 아니라 **이미 계산된 지표 값**(bands, week52,
  stoch, atrPercentValue, obvArray)을 인자로 받는 순수 함수로 설계 — deriveTickerData.js에
  이 v7 지표들을 실제로 채워 넣는 배선(스크리닝 UI와 실데이터 연결)은 아직 어느 스토리에도

  이번 스토리에서는 손대지 않음(범위 최소화). ATR 백분위는 percentileRank(value,population)
  = (value 이하 개수)/전체×100 방식으로 계산 — population=[1..10]일 때 value=3→30,
  value=7→70이 정확히 경계에 오는 성질을 이용해 경계 테스트를 구성함.

- ⚠ 다음 스토리(US-6) 착수 전 확인사항: deriveTickerData.js에 v7 지표(bollinger/week52/
  stochastic/atrPercent/obv) 필드가 아직 없다는 점을 인지할 것. US-6은 localStorage
  스키마 v2(필터 상태 모델)만 다루므로 이 갭과 무관하게 진행 가능하지만, US-7(화면1 UI)
  구현 시점에는 반드시 deriveTickerData.js 확장이 선행되어야 신규 필터가 실제로 동작함
