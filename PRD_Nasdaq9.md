# PRD: 나스닥 종목추천 v9 — 백테스트 검증 & 파라미터 조정

> **v8 → v9 변경점 (v8 PRD §11 예고 범위의 구체화):**
> 1. 🆕 **백테스트 엔진**: Node 스크립트가 **앱과 동일한 lib 코드를 그대로 import**해 과거 신호를 재현 — 로직 이중화 금지 (§4.1)
> 2. 🆕 **backtest.json**: 백테스트 결과를 정적 산출물로 발행, 클라이언트는 표시만 담당 — 기존 정적 아키텍처 유지 (§7)

> 4. 🆕 **과최적화 방지 장치**: 표본 내/표본 외(In/Out-of-Sample) 분할 검증, 자동 최적화 금지(운영자 승인 워크플로), 조정 이력 기록 (§4.4)
> 5. 🆕 **후보 변형 비교 프레임**: 이격도 역U자 점수·ADX·컨센서스 가중치를 "후보 변형"으로 구현 — **표본 외 성과가 우월할 때만 채택** (§4.3)
> 6. 🆕 **신뢰도 표시**: 화면2에 전략별 과거 승률·초과수익 표시 (면책·한계 고지 필수) (§4.5, §5)
> 7. 🆕 **한계 정직 고지**: 생존 편향, 펀더멘털 선견 편향, 평가 구간 길이 — 화면과 문서 양쪽에 명시 (§9)
> 8. 🔍 원칙 유지: 기존 동작 변경은 **백테스트 근거가 있을 때만**, 상수 조정은 `constants/v8.js` 단일 지점에서 changelog 주석과 함께

## 2. 배경 및 문제 정의

- v8의 배점·기준값(VCP 40/25/15/20, 펀더멘털 F1~F5 임계값, 완화 폴백 7/8, 컨센서스 등급)은 미너비니 원전과 설계 판단에 근거한 **설계값**이며 실측 검증이 없다. v8 화면에도 "v9 백테스트로 조정 예정"이라 고지했다.
- 컨센서스(★★) 종목이 단일 신호(★) 종목보다 실제로 더 나은지, 펀더멘털 Pass가 Partial/Fail보다 실제로 더 나은지가 미검증이다. 이 실측이 v9의 핵심 산출물이다.
- v8에서 이월된 추세추종 개선안(이격도 역U자, ADX)은 "그럴듯한 아이디어" 단계다. 기존 동작을 바꾸는 일이므로, 백테스트에서 우월함이 확인될 때만 채택한다.
- 백테스트의 3대 함정을 구조적으로 통제해야 한다: **① 로직 이중화**(백테스트용 재구현이 앱과 미묘하게 달라지는 문제 → 같은 lib를 import해 원천 차단), **② 과최적화**(표본 내 성과에 맞춘 파라미터가 미래에 무력 → In/Out 분할 + 자동 최적화 금지), **③ 선견·생존 편향**(§9에서 정직하게 고지하고 해석 지침 제공).

## 4. 핵심 기능 상세

### 4.1 백테스트 엔진 — "같은 코드" 원칙

- `scripts/backtest.mjs` (Node, 신규 의존성 없음)가 `src/lib/*`를 직접 import한다. 백테스트 전용 지표·판정 재구현은 **금지** — 앱이 실행하는 함수가 곧 백테스트가 실행하는 함수다
- 시점 재현: 평가일 t마다 "t까지의 데이터만" 잘라낸 시계열로 두 모드 1단계·2단계, 컨센서스, (가능 범위의) 펀더멘털 판정을 실행 — 미래 데이터 참조가 코드 구조상 불가능하도록 슬라이스 후 전달
- 완화 폴백(추세추종·미너비니 모두)도 당시 조건대로 재현하고, 폴백 적용 여부를 신호 레코드에 기록 (폴백 신호와 정상 신호의 성과를 분리 집계)
- 입력: `public/data/nasdaq100.json`(현행 스냅샷), `fundamentals.json`(있으면) / 출력: `public/data/backtest.json` + 콘솔 요약

### 4.2 평가 프로토콜

| 항목 | 정의 |
|---|---|
| 평가 스텝 | 5거래일 간격 (실제 운영이 주 1회 스냅샷이므로 주 단위가 운영 현실과 일치) |
| 평가 구간 | 데이터 시작 + 252거래일(지표 워밍업) 이후 ~ 최신 − 60거래일 (마지막 보유 기간 확보) |
| 보유 기간 | 5 / 20 / 60거래일 (진입 = 신호일 종가, 청산 = N거래일 후 종가 — 종가 기준 단순화, §9 고지) |
| 1차 성과 지표 | **초과수익** = 종목 수익률 − 같은 기간 유니버스(dataSufficient 전체) 등가중 평균 수익률 |
| 보조 지표 | 절대 수익률 평균·중앙값, 승률(초과수익 > 0 비율), 표본 수, (Should) 보유 중 MDD |
| 신호 정의 | 각 평가일의 "추천 상위 5종목" 기준 집계 + "1단계 통과 전체" 기준 집계 병행 (상위 추천의 성과와 신호 자체의 성과를 구분) |
| In/Out 분할 | 평가 구간 전반 50% = In-Sample(조정 근거), 후반 50% = Out-of-Sample(채택 판단·화면 표시용) |

**펀더멘털 축의 제약 (선견 편향 통제):** fundamentals.json은 현재 스냅샷이므로 과거 시점 판정에 그대로 쓰면 미래 정보를 참조하는 선견 편향이 생긴다. v9에서는 ① yfinance 분기 이력(약 4~5분기)으로 재구성 가능한 최근 구간만 펀더멘털 축을 평가하고, ② 그 이전 구간은 펀더멘털 축을 집계에서 제외하며, ③ backtest.json과 화면 모두에 "펀더멘털 축은 짧은 구간·근사 재구성 기반 참고치"임을 명시한다. 완전한 해결(포인트인타임 DB)은 Won't have.

### 4.3 후보 변형 비교 프레임 (Should)

- 변형 등록 인터페이스: 현행 로직 함수를 감싸거나 대체하는 순수 함수 + 메타(이름, 변경점 설명). 엔진이 현행과 변형을 같은 평가일·같은 데이터로 병렬 실행
- **채택 기준 (전부 충족 시에만):**
  1. Out-of-Sample 구간에서 보유 20거래일 기준 초과수익 평균과 승률이 **모두** 현행 이상
  2. Out 구간 표본 수 ≥ 100 신호 (미달 시 "판단 보류"로 기록)
  3. 운영자 승인 (근거 리포트 링크와 함께 constants changelog에 기록)
- 채택 시 처리: 변형 코드가 현행을 대체하되, **추세추종 모드 변경은 화면2에 "vN에서 백테스트 근거로 스코어링이 변경되었습니다" 안내 1줄** (기존 동작 변경 금지 원칙의 유일한 예외 — 근거·고지 동반 조건)
- 미채택 변형도 결과를 backtest.json `variants[]`에 남긴다 (기록이 곧 다음 개선의 출발점)

### 4.4 파라미터 조정 워크플로 (수동 승인 — 자동화 금지)

```
[1] backtest.mjs 실행 → 리포트 확인 (In/Out 분리 성과표)
[2] In-Sample 근거로 조정안 수립 (예: "VCP 변동성 수축 배점 25→30")
    — 한 번에 소수 항목만, 세밀 격자 탐색 금지 (과최적화 방지)
[3] constants/v8.js 수정 + changelog 주석 (날짜·변경 전후 값·근거 리포트 경로)
[4] backtest.mjs 재실행 → Out-of-Sample에서 현행 대비 악화 없음 확인
[5] npm test + build → 커밋 (backtest.json 갱신 포함)
```

- Out에서 악화되면 조정을 되돌리고 그 사실도 changelog에 남긴다 (실패 기록 = 자산)
- 이 워크플로는 README 운영 절차에 절로 추가한다

### 4.5 신뢰도 표시 (화면2)

- 모드별 헤더 요약 1줄: `최근 검증 구간(Out-of-Sample) 승률 n% · 유니버스 대비 +n.n%p (상위 5종목 · 20거래일 보유 기준)`
- 펼침 상세: 평가 기간, 표본 수, 보유 기간별(5/20/60) 성과표, 컨센서스 ★★ vs ★ 비교, (참고) 펀더멘털 축
- **고정 고지 문구 (요약 바로 아래, 생략 불가):** "과거 성과는 미래 수익을 보장하지 않습니다. 현재 구성 종목 기준 백테스트로 생존 편향이 존재하며, 거래 비용은 반영되지 않았습니다."
- 표시 데이터는 **Out-of-Sample 결과만** 사용 (In 성과 표시는 자기 채점이므로 금지)
- backtest.json 부재·스키마 불일치 시 신뢰도 영역 전체 미표시 (기존 화면과 동일 — graceful degradation)

- [ ] 4개 전략 축 × In/Out × 보유 3종 성과표가 backtest.json으로 발행되고 스키마 검증을 통과한다
- [ ] 컨센서스 ★★ vs ★, 펀더멘털 Pass/Partial/Fail의 실측 비교표가 산출된다 (v8 설계 가설의 검증)

- [ ] (변형 채택 시) 채택 근거가 Out 기준 충족을 만족하고 변경 안내가 화면에 표시된다 / (미채택 시) variants[]에 결과가 기록된다
- [ ] 기존 테스트 전체 통과 — 백테스트 도입 자체는 앱 기존 동작을 일절 바꾸지 않는다

## 9. 한계 고지 (문서·화면 공통 — 정직성 원칙)

1. **생존 편향**: 현재 나스닥100 구성 종목 기준이므로 과거에 편출된 종목이 빠져 성과가 낙관 편향될 수 있다
2. **펀더멘털 선견 편향**: 펀더멘털 축은 분기 이력 근사 재구성 기반의 짧은 구간 참고치다
3. **거래 비용 미반영**: 수수료·세금·슬리피지 미포함, 종가 진입·청산 단순화
4. **평가 구간 길이**: 2년(3년 확대 시에도) 데이터는 단일 시장 국면에 치우칠 수 있다 — 강세장 데이터로 검증된 로직이 약세장에서 동일하게 작동한다는 보장 없음
5. **과최적화 잔존 위험**: In/Out 분할로 완화하지만 제거는 불가 — 그래서 조정은 보수적으로, 소수 항목만
