# prd-v9.1-diagnostics.md — 백테스트 심화 진단 (Ralph 호환 태스크 문서)

> 배경: v9 백테스트 1차 해석(2026-07)에서 세 가지 검증 가설이 도출됐다.
> ① In 구간 추세추종 붕괴의 범인이 완화 폴백 신호인가 (relaxedShare In 30.9% vs Out 18.6%)
> ② 청산 규칙(손절 −8% + 시간/트레일링)이 60일 엣지를 지키며 MDD 16~18% 꼬리를 자를 수 있는가
> ③ 신호 신선도 프리미엄이 실재하는가 — 매일 운영 전환 여부를 결정할 측정
> **v9.1은 측정 릴리스다. 운영 주기·파라미터·추천 로직은 아무것도 바꾸지 않는다.**
> 루프 규칙: 매 이터레이션마다 미완료 스토리 중 가장 위의 것 하나만 구현하고,
> 승인 기준을 전부 통과하면 체크박스를 채우고 커밋한 뒤 이터레이션을 종료한다.

## 프로젝트 컨텍스트 (매 이터레이션 필독)

- v9 자산 위에서 작업: `scripts/backtest.mjs`, `scripts/lib/{asOf,performance,variants,fundamentalHistory}.mjs`,
  `src/lib/backtestSchema.js`, `src/lib/backtestLoader.js`, 화면2 신뢰도 표시
- 절대 원칙 (v9와 동일 + v9.1 추가):
  - **같은 코드 원칙**: 판정·스코어링은 `src/lib/*` import만. 재구현 금지
  - **미래 참조 불가**: 슬라이스 후 전달 구조 유지. 청산 규칙(US-2)은 보유 구간 내
    일별 종가만 사용 — 신호일 이후 데이터가 "진입 판정"에 역류하지 않아야 함
  - **자동 파라미터 최적화 금지 / adopted는 항상 false**: 측정·기록까지만. 청산 규칙·운영 주기의
    채택 결정은 운영자 몫
  - **constants/v8.js·src 추천 로직 수정 금지**: 변형·청산 규칙 코드는 scripts/ 아래에만
  - **Out-of-Sample만 화면 표시**, backtest.json 부재 시 graceful degradation 유지
  - 새 npm 의존성 금지, 기존 테스트 전체 통과 유지
- 품질 게이트 (모든 스토리 공통): `npm run test` 전체 통과 + `npm run build` 성공

## 유저 스토리 (우선순위 순 — 위에서부터 하나씩)

### US-1. backtest.json 스키마 v2 — 완화/정상 분리 집계
- [x] **운영자로서, 완화 폴백(7/8·완화 프리셋) 신호와 정상 신호의 성과를 분리해 보고 싶다.**

**구현:** `scripts/backtest.mjs` + `src/lib/backtestSchema.js` + `src/lib/backtestLoader.js`
- 전략 레코드에 차원 추가: `signalQuality: "all" | "normal" | "relaxed"` — 기존 집계는
  `"all"`로 유지하고, normal/relaxed 레코드를 추가 발행 (기존 신호 레코드에 이미
  relaxationApplied가 있으므로 집계 축만 추가)
- `schemaVersion: 2`. 검증기는 v2 규칙 적용, **로더는 v1 파일도 하위 호환**으로 수용
  (signalQuality 부재 → "all"로 정규화 — research.json v1→v2 때의 패턴 재사용)
- 화면 표시는 기존대로 `"all"` 레코드만 사용 (분리 집계는 운영자 분석용 — UI 확장은 이번 범위 밖)
- 콘솔 요약에 In/Out × normal/relaxed 비교표 출력 (가설 ① 판정 재료)

**승인 기준:**
1. normal 신호 수 + relaxed 신호 수 = all 신호 수 (전 전략·전 보유 기간, 테스트로 보증)
2. v1 픽스처가 로더에서 하위 호환으로 정상 표시된다 + 무효 v2 픽스처(signalQuality enum 오류) 거부
3. 화면2 렌더링이 v9와 동일하다 (all만 사용 — 기존 UI 테스트 전부 통과)

### US-2. 변형 D — 청산 규칙 (경로 의존 성과 계산)
- [x] **운영자로서, 손절·트레일링 청산이 고정 보유 대비 성과와 MDD를 어떻게 바꾸는지 측정하고 싶다.**

**구현:** `scripts/lib/exits.mjs` 신규 + `scripts/lib/performance.mjs` 확장 + variants 등록
- performance에 경로 의존 청산 지원 추가: 보유 구간의 **일별 종가 경로**를 순회하며
  청산 조건 판정 (기존 고정 N일 청산은 변경 없이 공존 — 기본 경로)
- 변형 2종만 등록 (격자 자제 원칙):
  - `exit_stop8_time60`: 진입 종가 대비 종가 −8% 도달 시 익일이 아닌 **당일 종가 청산**,
    미도달 시 60거래일 청산
  - `exit_stop8_trail15`: 위 손절 + 보유 중 최고 종가 대비 −15% 이탈 시 당일 종가 청산,
    미도달 시 60거래일 청산
- 한계 명시 (레코드 note 고정 문구): "종가 기준 판정 — 장중 이탈 미반영으로 실제 손절 체결가는
  이보다 불리할 수 있음"
- 비교 기준: 현행 60거래일 고정 보유 (같은 신호 집합 — 진입은 동일, 청산만 상이)
- variants[] 레코드 확장: `outDetail: { winRate, avgExcess, medianExcess, avgReturn, mdd,
  avgHoldingDays, stopHitRate }` (스키마 v2에 optional 필드로 추가)
- **adopted는 false 고정.** 판정 재료(60일 엣지 보존 여부 + MDD 개선 폭)를 note에 서술

**승인 기준:**
1. 손계산 검증: 소형 픽스처(가격 경로 3종 — 손절 도달/트레일링 도달/만기 보유)에서
   청산일·청산가·수익률이 수작업 기대값과 일치 — 6개 이상
2. 손절 판정에 신호일 이전 데이터·경로 밖 데이터가 개입 불가함이 테스트로 보증된다
3. avgHoldingDays·stopHitRate가 집계되고, note 고정 문구(종가 기준 한계)가 존재한다
4. src/ 미변경 + adopted 전부 false

### US-3. stepDays 파라미터화 + 겹침 보정 명시
- [ ] **운영자로서, 매일 평가(stepDays=1) 시뮬레이션을 실행하되 표본 착시 없이 읽고 싶다.**

**구현:** `scripts/backtest.mjs` CLI 확장
- `--step=N` 인자 (기본 5, 허용 1~10). config.stepDays에 반영
- **겹침 보정 명시**: config에 `overlapFactor` 추가 = 보유 기간별 `holdingDays / stepDays`,
  콘솔 요약과 backtest.json 양쪽에 "명목 표본 n / 유효 독립 표본 근사 n÷overlapFactor" 병기
- step=1 실행은 연산량이 5배 — 진행률 로그 추가 (평가일 단위)
- 산출 파일 충돌 방지: `--out=경로` 인자 (기본 public/data/backtest.json).
  step≠5 실험 산출물은 기본 경로에 쓰지 않도록 **step≠5이면 --out 필수** (화면 표시용
  공식 파일은 항상 step=5 산출물이라는 운영 규칙 — README에 명기)

**승인 기준:**
1. `--step=1 --out=...` 실행이 픽스처에서 완주하고 평가일 수가 step=5의 약 5배다
2. step≠5에서 --out 누락 시 실행 거부 (공식 파일 보호 테스트)
3. overlapFactor가 config에 기록되고 콘솔에 유효 표본 근사가 병기된다
4. step=5 기본 실행 산출물이 v9.1 스키마 하에서 기존과 동일 구조다 (회귀 없음)

### US-4. 신호 신선도 코호트 분석
- [ ] **운영자로서, 신선한 신호가 실제로 더 나은지 — 매일 운영 전환의 근거 — 를 측정하고 싶다.**

**구현:** `scripts/lib/freshness.mjs` 신규
- 신선도 = 신호일 − 트리거 이벤트 발생일 (거래일 기준). 이벤트 정의 (모드별, 문서와 코드 주석에 명기):
  - 추세추종: 가장 최근 **골든크로스 발생일** (기존 lib의 골든크로스 판정 재사용 — 재구현 금지)
  - 미너비니: 종가가 직전 63거래일 최고 종가를 상향 돌파한 **가장 최근일** (피벗 돌파 근사).
    돌파 이력이 최근 10거래일 내에 없으면 "no_recent_breakout" 코호트
- 코호트: `0d / 1-2d / 3-4d / 5d+ / no_recent_breakout` — 전 신호가 정확히 하나의 코호트에 귀속
- backtest.json v2에 `freshnessCohorts[]` 추가: 전략 × sample × 코호트 × byHolding
  (basis는 allSignals만 — top5는 코호트별 표본이 너무 작아짐)
- 콘솔 요약: Out 구간 코호트별 20일·60일 초과수익 표 (가설 ③ 판정 재료)
- step=1 실행과 조합 시 신선도 해상도가 올라감을 README 분석 절에 기록

**승인 기준:**
1. 코호트 귀속의 완전성·배타성 테스트 (전 신호 합 = 전체, 중복 귀속 없음)
2. 이벤트 판정이 기존 lib 함수 재사용임이 확인된다 (골든크로스 재구현 없음)
3. 픽스처 기반: 이벤트일을 알고 있는 합성 케이스에서 코호트 배정이 기대값과 일치 — 5개 이상
4. 스키마 v2 검증기가 freshnessCohorts를 검증하고, 부재 시(구버전 파일) 로더가 정상 동작한다

### US-5. (Should) 화면2 신뢰도 표시 — 20일·60일 병기
- [ ] **사용자로서, 대표 숫자 하나가 아니라 단기(20일)와 검증된 보유 지평(60일)을 함께 보고 싶다.**

**구현:** 화면2 신뢰도 요약 확장
- 요약 1줄을 2지평 병기로: `검증 구간 초과수익 — 20거래일 +n.n%p · 60거래일 +n.n%p
  (★★ 컨센서스 · 상위 5종목 기준)` — 통합 뷰의 대표 축을 consensus_2star로
- 60일 숫자 옆에 유효 표본 주석: "겹침 보정 유효 표본 약 n건" (config.overlapFactor 사용)
- 고정 고지 문구(생존 편향·비용 미반영·과거≠미래)는 그대로 유지, 문구 위치 불변
- v1 backtest.json(overlapFactor 없음) 로드 시 유효 표본 주석만 생략 (하위 호환)

**승인 기준:**
1. 2지평 병기 렌더링 + Out 전용 규칙 유지 테스트
2. overlapFactor 부재 시 주석 생략 하위 호환 테스트
3. 고지 문구 동반 렌더링 기존 테스트 유지
