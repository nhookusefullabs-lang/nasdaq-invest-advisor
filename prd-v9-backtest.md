# prd-v9-backtest.md — 백테스트 검증 & 파라미터 조정 (Ralph 호환 태스크 문서)

> 원본 설계: `PRD_Nasdaq9.md` (v9). 이 문서는 Ralph 루프가 한 이터레이션에 스토리 하나씩
> 구현→검증→커밋하도록 변환한 실행용 문서다.
> 루프 규칙: 매 이터레이션마다 **미완료 스토리 중 가장 위의 것 하나만** 골라 구현하고,
> 승인 기준을 전부 통과하면 체크박스를 채우고 커밋한 뒤 이터레이션을 종료한다.

## 프로젝트 컨텍스트 (매 이터레이션 필독)

- 기존 시스템: React+Vite+Tailwind 정적 SPA, GitHub Pages 배포, v8(이중 모드) 운영 중
- 추천 로직은 전부 `src/lib/*` 순수 함수: recommend.js(추세추종), minervini.js, consensus.js,
  fundamentals.js, indicators.js, constants/v8.js — **v9에서 이들 로직 수정 금지**
  (허용되는 유일한 변경: Node에서 import 가능하도록 export 구문 보강 — 동작 불변)
- 테스트: vitest (**기존 테스트 전체 통과 상태 유지 필수 — 깨뜨리면 해당 이터레이션 실패**)
- v8 산출 자산 재사용: 2년치 합성 픽스처(`fixtures/nasdaq100.2y.sample.json`),
  fundamentals 픽스처, researchLoader의 graceful degradation 패턴, 원자적 쓰기 패턴
- 절대 원칙:
  - **같은 코드 원칙**: 백테스트는 `src/lib/*`를 직접 import한다. 지표·판정·스코어링의
    백테스트 전용 재구현은 어떤 이유로도 금지 (발견 시 해당 이터레이션 실패)
  - **미래 참조 불가**: 평가일 t의 판정 함수에는 t 이후 데이터가 물리적으로 전달되지 않아야 한다
    (슬라이스 후 전달 구조 — 테스트로 보증)
  - **자동 파라미터 최적화 금지**: 루프는 측정·기록·표시까지만 한다. constants/v8.js의
    값 변경, 변형(variant)의 채택 결정은 운영자 몫 — 루프가 상수를 바꾸면 이터레이션 실패
  - **Out-of-Sample만 표시**: 화면에 노출되는 성과는 `sample:"out"` 레코드만
  - backtest.json 부재 시 앱 완전 정상 동작 (graceful degradation)
  - 새 npm 의존성 추가 금지
- 품질 게이트 (모든 스토리 공통): `npm run test` 전체 통과 + `npm run build` 성공

## 유저 스토리 (우선순위 순 — 위에서부터 하나씩)

### US-1. 백테스트 부트스트랩 — Node에서 앱 lib 그대로 실행
- [x] **개발자로서, 백테스트 스크립트가 앱과 동일한 로직 코드를 import해 실행할 수 있는 기반을 원한다.**

**구현:** `scripts/backtest.mjs` 스캐폴드
- Node ESM으로 `src/lib/*` 직접 import 확인 — 안 되는 모듈은 export 구문만 보강 (로직·시그니처 불변,
  기존 테스트로 불변성 검증). 브라우저 전용 API(window 등) 의존이 lib에 있으면 해당 의존을
  주입 가능한 인자로 분리하되 앱 쪽 호출부 동작 불변 유지
- 스캐폴드 실행 흐름: 데이터 로드(경로 인자, 기본 `public/data/nasdaq100.json`) → 스모크:
  최신 시점 기준 두 모드 추천을 각 1회 실행하고 결과 요약을 콘솔 출력 → exit 0
- 스모크 결과가 앱과 동일함을 보증하는 테스트: 같은 픽스처로 vitest에서 lib 직접 호출한 결과와
  backtest.mjs 내부 호출 결과가 일치 (동형성 테스트)

**승인 기준:**
1. `node scripts/backtest.mjs fixtures/nasdaq100.2y.sample.json` 실행이 성공하고 두 모드 요약을 출력한다
2. 백테스트 쪽에 지표·판정 재구현 코드가 없다 (import만 존재 — 리뷰 체크 항목으로 progress.txt에 기록)
3. lib export 보강이 있었다면 기존 테스트 전체 통과로 동작 불변이 확인된다

### US-4. 성과 계산 — 초과수익·승률·표본
- [x] **운영자로서, 각 신호가 5/20/60거래일 후 유니버스 평균 대비 얼마나 벌었는지 알고 싶다.**

**구현:** `scripts/lib/performance.mjs`
- 신호별: 진입 = 신호일 종가, 청산 = N거래일 후 종가 (N=5/20/60), 수익률 계산.
  청산일이 데이터 범위를 벗어나는 신호는 해당 보유 기간 집계에서 제외 (표본 수에 반영)
- 벤치마크: 같은 진입~청산 구간의 **유니버스(dataSufficient 전체) 등가중 평균 수익률** —
  초과수익 = 종목 수익률 − 벤치마크
- 집계: 전략 축(trend/minervini/consensus_2star/consensus_1star) × basis(top5/allSignals) ×
  보유 기간별 → 승률(초과수익>0 비율)·평균/중앙 초과수익·평균 절대수익·표본 수·relaxedShare
- (Should) 보유 중 MDD: 진입~청산 구간 종가 기준 최대 낙폭 평균
- 부동소수 처리: 집계값은 소수 4자리 반올림 (JSON 안정성)

**승인 기준:**
1. 손계산 검증 테스트: 소형 픽스처(3종목·60거래일)로 수익률·벤치마크·초과수익·승률을
   수작업 기대값과 대조 — 6개 이상
2. 경계 테스트: 청산일 범위 초과 제외, 표본 0인 축의 안전 처리(NaN 금지, null 표기)
3. ★★와 ★가 분리 집계된다 (컨센서스 검증이라는 v9 핵심 목적)

### US-7. 후보 변형 비교 프레임 + 변형 A/B/C
- [x] **운영자로서, 개선 아이디어(역U자 이격도·ADX·컨센서스 가중)를 현행과 같은 잣대로 비교한 기록을 원한다.**

**구현:** `scripts/lib/variants.mjs` + 변형 3종
- 변형 등록 구조: `{ name, description, apply }` — apply는 현행 lib 함수를 감싸거나 대체하는
  순수 함수 (앱 코드 수정 없음 — 변형 로직은 scripts/ 아래에만 존재)
- 변형 A `disparity_inverted_u`: 이격도 점수를 역U자(+3~+8% 만점, 과열 감점)로 바꾼 추세추종 2단계
- 변형 B `adx_gate`: ADX(14) ≥ 20을 추세추종 1단계 추가 게이트로 (ADX 계산은 scripts 측 유틸 —
  앱 lib에 넣지 않음. 채택 확정 시에만 별도 이터레이션에서 앱 반영)
- 변형 C `consensus_weighted`: 백분위 가중 평균 3격자(50:50 현행 / 60:40 / 40:60) — 세밀 격자 금지
- 엔진이 현행+변형을 같은 평가일·같은 슬라이스로 병렬 실행 → variants[]에
  `{ name, adopted:false, outVsBaseline:{avgExcessDelta, winRateDelta}, note }` 기록
- **채택 결정·상수 변경·앱 반영은 루프가 하지 않는다** — adopted는 항상 false로 기록,
  판단 재료(Out 표본 수 ≥100 충족 여부 포함)를 note에 서술

**승인 기준:**
1. 변형 3종이 등록·병렬 실행되고 variants[]에 델타가 기록된다 (픽스처 기반)
2. 변형 코드가 scripts/ 밖(src/)에 존재하지 않고, constants/v8.js가 변경되지 않았다
3. 변형 A·B의 신호가 현행과 실제로 달라지는 픽스처 케이스가 테스트로 존재한다
   (변형이 무의미하게 동일 결과만 내는 구현 방지)
4. adopted가 전부 false다

### US-8. 신뢰도 표시 UI — Out-of-Sample만, 고지 동반
- [x] **사용자로서, 각 전략의 검증 구간 성과를 한계 고지와 함께 화면2에서 확인하고 싶다.**

**승인 기준:**

3. backtest.json 부재 시 v8 화면과 동일 (graceful degradation 테스트) + 기존 테스트 전부 유지
4. 표본 0/null 축의 표시가 안전하다 ("표본 부족" 표기, NaN 노출 금지)
