# prd-v7-filters-presets.md — 지표 필터 확장 + 추천 프리셋 (Ralph 호환 태스크 문서)

> 원본 설계: `PRD_Nasdaq7.md` (v7). 한 이터레이션에 스토리 하나씩 구현→검증→커밋한다.
> 루프 규칙: 미완료 스토리 중 **가장 위의 것 하나만** 구현. 승인 기준 전부 통과 시에만
> 체크박스를 채우고 커밋 후 이터레이션 종료. 막히면 체크하지 않고 progress.txt에 기록 후 종료.

## 프로젝트 컨텍스트 (매 이터레이션 필독)

- 기존 시스템: React+Vite+Tailwind 정적 SPA, `public/data/nasdaq100.json` 스냅샷, vitest
- **최상위 회귀 원칙**: "기본형 프리셋 + 신규 필터 전부 꺼짐" 상태에서 모든 화면의 결과가
  v5(현행)와 완전히 동일해야 한다. 이 원칙을 깨는 커밋은 실패다
- 기존 필터 4종의 판정 로직 변경 금지 (UI 배치 변경만 허용)
- 신규 지표 5종은 화면1 필터 전용 — 추천 스코어링(2단계 배점)에 절대 편입하지 않는다
- 새 npm 의존성 추가 금지 (필요 시 중단 후 progress.txt에 사유 기록)
- 품질 게이트 (공통): `npm run test` 전체 통과 + `npm run build` 성공
- **선행 조건**: v6 리서치 서브에이전트 루프(prd-research-agent.md)가 완료된 상태에서 시작
  (recommend.js를 양쪽 루프가 동시에 건드리는 충돌 방지)

## 유저 스토리 (우선순위 순 — 위에서부터 하나씩)

### US-1. 데이터 수집 스크립트 12개월 전환
- [x] **운영자로서, 52주 지표 계산이 가능하도록 12개월치 데이터를 수집하고 싶다.**

**구현:** `scripts/collect_data.py` 수정
- `MONTHS_BACK = 6 → 12`, `MIN_TRADING_DAYS = 90 → 200`
- 그 외 로직 변경 금지 (스키마·출력 경로 동일)
- 스크립트 상단 주석에 "12개월 수집 — 52주 신고가/신저가 지표용" 명시

**승인 기준:**
1. 두 상수가 변경되고 다른 로직 diff가 없다
2. **루프는 이 스크립트를 실행하지 않는다** (재수집·재배포는 운영자 수동 작업 — progress.txt에
   "운영자 액션 필요: collect_data.py 재실행 후 JSON 커밋" 기록)

### US-2. sixMonth 차트 정의 변경 (수집 기간 변경 파급효과 차단)
- [x] **사용자로서, 12개월 데이터가 들어와도 6개월 미니 차트는 계속 6개월이길 원한다.**

**구현:** `deriveTickerData()`의 `chart.sixMonth`를 "수집된 전체 기간" → **"최근 126거래일"**로 변경
- 데이터가 126거래일 이하면 전체 기간 사용 (현행 6개월 데이터와 하위 호환)

**승인 기준:**
1. 200일짜리 시리즈 입력 시 sixMonth 길이가 126이다 (테스트)
2. 120일짜리 시리즈 입력 시 sixMonth가 전체(120)와 같다 — 현행 데이터 회귀 없음 (테스트)
3. oneMonth(21)·threeMonth(63) 기존 테스트 전부 유지

### US-3. 신규 지표 함수 1차 — 볼린저밴드 + 52주 신고가/신저가
- [x] **개발자로서, 볼린저밴드와 52주 고저를 순수 함수로 계산하고 싶다.**

**구현:** `src/lib/indicators.js`에 추가 (기존 함수 수정 금지)
- `bollingerBands(series, period=20, mult=2)` → {middle, upper, lower} (σ = 표본표준편차)
- `week52HighLow(series)` → {high, low} 또는 **데이터 252거래일 미만 시 null**

**승인 기준:**
1. 수기 계산 대조 테스트 각 2개 이상 (알려진 소규모 시리즈로 기대값 검증)
2. 252일 미만 입력 시 null 반환 테스트
3. vitest 테스트 6개 이상 추가, 기존 테스트 유지

---

### US-4. 신규 지표 함수 2차 — 스토캐스틱 + ATR + OBV
- [ ] **개발자로서, 스토캐스틱·ATR·OBV를 순수 함수로 계산하고 싶다.**

**구현:** `src/lib/indicators.js`에 추가
- `stochastic(series, 14, 3, 3)` → {slowK, slowD} — 분모 0(14일 고저 동일)이면 해당일 null
- `atr(series, 14)` → Wilder 평활 ATR. 보조 `atrPercent(series)` = ATR/종가×100
- `obv(series)` → 누적 OBV 배열 (상승일 +V, 하락일 −V, 보합 0)

**승인 기준:**
1. 수기 계산 대조 테스트 각 2개 이상 (특히 ATR Wilder 평활 초기값, OBV 보합일)
2. 스토캐스틱 분모 0 케이스 null 처리 테스트
3. vitest 테스트 8개 이상 추가

### US-5. 필터 판정 로직 + 임계 상수
- [ ] **사용자로서, 신규 지표 5종으로 종목을 걸러내고 싶다.**

**구현:** `src/lib/constants.js`(임계 상수) + `src/lib/filters.js`(판정 함수)
- 상수: 52주 근접 ±5%, 볼린저 하단 근접 계수 1.02, 스토캐스틱 20/80, ATR 백분위 30/70, OBV SMA 창 20
- 판정 함수 (필터당 옵션 2종, PRD §3 Must 1~5 정의 그대로):
  볼린저(하단 근접/상단 돌파), 52주(신고가 근접/신저가 근접), 스토캐스틱(과매도/과매수),
  ATR(저변동성/고변동성 — **유니버스 dataSufficient 종목의 ATR% 백분위** 기준), OBV(상승/하락 추세)
- 52주 필터: 252일 미만 종목은 제외하되 **제외 종목 수를 반환** (UI 안내용)

**승인 기준:**
1. 필터×옵션 10케이스 각각 통과/탈락 테스트
2. ATR 백분위 경계(정확히 30%/70% 지점) 테스트
3. 52주 필터 제외 카운트 반환 테스트
4. vitest 테스트 12개 이상 추가

### US-6. 필터 상태 모델 + localStorage 스키마 v2
- [ ] **사용자로서, 신규 필터와 프리셋 설정이 새로고침 후에도 유지되길 원한다.**

**구현:** 필터 state에 신규 5종 추가(기본 꺼짐) + `storage.js` 스키마 버전 1→2
- v2 신규 필드: 신규 필터 상태, `preset`('conservative'|'default'|'aggressive'|'custom'), `customParams`,
  `researchRequests`(티커 배열 — US-11 리서치 요청 목록용, 기본 빈 배열)
- **마이그레이션**: v1 로드 시 기존 필드(선택 종목, 비중, 기존 필터) 보존 + 신규 필드 기본값
- 복원 시 customParams 범위 검증 (RSI 30~70, 골든크로스 1~20, 편입임계 50~95) — 범위 밖이면 해당 값만 기본형으로
- 복원 시 researchRequests에서 유니버스에 없는 티커는 조용히 제거 (기존 선택 종목 정리 패턴과 동일)

**승인 기준:**
1. v1 데이터 → v2 마이그레이션에서 선택 종목·비중·기존 필터 보존 테스트
2. 범위 밖 customParams 복원 시 기본형 리셋 테스트
3. 신규 필터 초기값 전부 꺼짐 테스트

### US-8. recommend.js 설정 객체화 + 프리셋 정의
- [ ] **개발자로서, 추천 파라미터를 설정 객체로 주입해 프리셋 전환이 가능하게 하고 싶다.**

**구현:** `src/lib/presets.js` 신설 + `recommend.js` 리팩터링
- 프리셋 3종 (PRD §3 표): 보수형(RSI≥55/골든크로스 3일/편입 80), 기본형(50/5/70=현행), 공격형(45/10/60)
- recommend 함수가 설정 객체를 인자로 받도록 변경, 미전달 시 기본형
- 2단계 배점(60/30/10)은 설정 대상 아님 — 고정 유지

**승인 기준:**
1. **기본형으로 실행한 추천 결과가 리팩터링 전과 완전 동일** (핵심 회귀 — 실데이터 스냅샷 비교 테스트)
2. 보수형/공격형 각각 파라미터가 반영되는 테스트 (경계 종목 픽스처)
3. 기존 추천 테스트 전부 유지 (v6 US-1 CLI도 여전히 기본형으로 동일 결과)

### US-10. 고급 설정 패널 (자유 파라미터)
- [ ] **고급 사용자로서, 프리셋 파라미터 3종을 직접 조정하고 싶다.**

**구현:** 프리셋 세그먼트 아래 접이식 "고급 설정" 패널 (기본 접힘)
- RSI 하한/골든크로스 창/편입 임계 3개 입력, 허용 범위 강제 (30~70 / 1~20 / 50~95)
- 프리셋 값과 달라지는 순간 세그먼트가 "사용자 설정"으로 전환, 프리셋 버튼 클릭 시 해당 값으로 덮어씀
- 각 입력 옆 프리셋 대비 변경 표시, "기본형으로 초기화" 버튼
- customParams는 US-6 스키마로 저장/복원

**승인 기준:**
1. 범위 밖 입력 차단 테스트 (3개 파라미터 각 상/하한)
2. 조정 → "사용자 설정" 전환 → 프리셋 클릭 → 값 덮어씀 흐름 테스트
3. 초기화 버튼 동작 테스트
4. 패널 기본 접힘 + 열지 않은 사용자 경로에 영향 없음 (회귀)

### US-11. 리서치 요청 목록 (v6 연계, 절충안 A)

**구현:** `src/components/ResearchRequestList.jsx` + 종목 카드/행의 요청 토글 버튼
- 화면1(검색 결과 행), 화면2(추천 카드), 화면3(종목 카드)에 "리서치 요청" 토글 버튼
  — 담김/해제 상태 표시, 유니버스 내 종목만 대상
- 공통 접이식 패널 (모든 화면에서 접근 가능, 기본 접힘): 담긴 티커 목록, 개수 배지,
  개별 제거(✕), 전체 비우기, **"목록 복사" 버튼**
- 복사 형식: 쉼표+공백 구분 티커 문자열 (예: `AAPL, MRNA, AXON`) — v6 리서치 커맨드의

- 복사 성공 시 "복사됨" 피드백 표시 (클립보드 API 실패 시 문자열을 선택 가능한 형태로 노출하는 폴백)
- 중복 담기 방지 (동일 티커 재클릭 = 해제), 상태는 US-6의 `researchRequests` 필드로 저장/복원
- **리서치를 실행하는 기능이 아님** — 목록 수집·전달까지만. 실행 트리거·API 호출 코드를 추가하지 않는다

**승인 기준:**
1. 토글 담기/해제, 중복 방지, 개별 제거, 전체 비우기 테스트
2. 복사 문자열이 `TICKER, TICKER` 형식과 일치하는 테스트 (클립보드는 목킹)
3. 새로고침(스토리지 복원) 후 목록 유지 + 유니버스 밖 티커 제거 테스트
4. 개수 배지 정확성, 빈 목록일 때 패널 상태 테스트
5. 세 화면 모두에서 버튼 렌더링 확인, 화면4는 변경 없음 (회귀)
